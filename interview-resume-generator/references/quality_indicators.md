# 回答质量判断标准

本文档定义了如何评估用户回答的质量，以及相应的追问策略。

---

## 评分体系

### 总分：10分

| 维度 | 分值 | 权重 | 说明 |
|-----|------|------|------|
| 具体性 | 0-3分 | 30% | 回答是否具体、明确，不是泛泛而谈 |
| 量化程度 | 0-3分 | 30% | 是否包含数字、比例、具体数据 |
| 完整性 | 0-2分 | 20% | 是否回答了问题的核心要点 |
| 亮点度 | 0-2分 | 20% | 是否有独特价值、突出成就 |

---

## 维度详解

### 1. 具体性（0-3分）

| 分数 | 标准 | 示例 |
|-----|------|------|
| 0分 | 完全不具体，只有模糊概念 | "我负责开发"、"做了很多事" |
| 1分 | 有基本方向，但不够具体 | "负责前端开发"、"做了几个项目" |
| 2分 | 有具体内容，但细节不足 | "负责抖音电商首页的前端开发" |
| 3分 | 非常具体，有清晰场景 | "负责抖音电商C端首页，使用React+TypeScript，带领5人团队" |

**判断要点**：
- 是否有明确的产品/模块名称？
- 是否有具体的技术栈？
- 是否有清晰的角色定义？
- 是否有团队规模说明？

### 2. 量化程度（0-3分）

| 分数 | 标准 | 示例 |
|-----|------|------|
| 0分 | 完全没有数据 | "效果很好"、"提升了很多" |
| 1分 | 有模糊程度词 | "大幅提升"、"显著改善" |
| 2分 | 有具体数字但不够精确 | "提升了50%左右"、"处理了几百万数据" |
| 3分 | 有精确的量化数据 | "首屏加载时间从3.2s优化到1.1s，提升66%"、"日处理订单从10万提升到100万" |

**判断要点**：
- 是否有百分比？
- 是否有绝对数值？
- 是否有对比（前后对比）？
- 是否有时间范围？

### 3. 完整性（0-2分）

| 分数 | 标准 | 示例 |
|-----|------|------|
| 0分 | 只回答了问题的一小部分 | 问：负责什么、技术栈、成果？答：负责开发 |
| 1分 | 回答了大部分内容 | 回答了负责什么和技术栈，但缺少成果 |
| 2分 | 完整回答了所有要点 | 负责什么、技术栈、成果都有涉及 |

**判断要点**：
- 是否回答了问题的所有子问题？
- 是否覆盖了STAR法则的各个维度？
- 是否有遗漏的关键信息？

### 4. 亮点度（0-2分）

| 分数 | 标准 | 示例 |
|-----|------|------|
| 0分 | 没有突出亮点 | "正常完成工作任务" |
| 1分 | 有一定亮点但不够突出 | "按时完成项目，获得团队认可" |
| 2分 | 有非常突出的亮点 | "主导的方案被公司推广到全集团使用"、"技术创新申请了专利" |

**判断要点**：
- 是否有超越常规的表现？
- 是否有独特的技术创新？
- 是否有重大业务影响？
- 是否获得重要认可？

## 质量等级与响应策略

### 总分 0-3 分：低质量 → 必须追问

**特征**：
- 回答过于简短（少于10个字）
- 只有模糊概念，没有具体内容
- 明显在敷衍

**响应策略**：
1. 肯定用户的回答（不要批评）
2. 提供具体的示例引导
3. 分解问题，一次只问一点

**示例对话**：
```
用户：我负责开发
助手：了解！能具体说说你负责哪个产品或模块吗？比如是APP、网站还是后台系统？
```

### 总分 4-6 分：中等质量 → 适度追问

**特征**：
- 有基本内容，但细节不足
- 缺少量化数据
- 亮点不够突出

**响应策略**：
1. 肯定已有信息
2. 针对缺失部分追问
3. 提供思考方向

**示例对话**：
```
用户：我负责抖音电商首页的前端开发，用了React
助手：不错！这个项目有没有什么成果数据？比如用户量、性能提升、业务增长等？
```

### 总分 7-8 分：良好质量 → 简单确认

**特征**：
- 内容具体
- 有一定量化
- 表达清晰

**响应策略**：
1. 确认关键信息
2. 小范围补充
3. 记录亮点

**示例对话**：
```
用户：负责抖音电商首页，React技术栈，将首屏加载从3s优化到1.2s
助手：太棒了！这个性能优化数据很有说服力。还有其他想补充的吗？比如团队规模、技术难点等
```

### 总分 9-10 分：优秀质量 → 深度挖掘

**特征**：
- 内容非常具体
- 量化数据精确
- 有突出亮点

**响应策略**：
1. 大力肯定
2. 深挖更多细节
3. 作为简历核心卖点

**示例对话**：
```
用户：带领8人团队重构抖音电商支付系统，日订单处理能力从10万提升到100万，方案被推广到全集团使用
助手：非常出色！这个经历非常亮眼，建议作为简历的核心卖点。能再详细说说重构过程中遇到的最大技术挑战是什么吗？怎么解决的？
```

## 不同场景的质量判断

### 社招场景

**重点维度**：量化程度、亮点度

**理由**：社招需要展示工作成果，量化数据和亮点是核心竞争力

**调整权重**：
- 具体性：25%
- 量化程度：35%
- 完整性：15%
- 亮点度：25%

### 校招场景

**重点维度**：具体性、完整性

**理由**：校招没有太多工作经验，重点看学习能力和项目经历的具体程度

**调整权重**：
- 具体性：35%
- 量化程度：20%
- 完整性：30%
- 亮点度：15%

### 转行场景

**重点维度**：完整性、亮点度

**理由**：转行需要展示可迁移技能和学习意愿，完整性更重要

**调整权重**：
- 具体性：25%
- 量化程度：20%
- 完整性：35%
- 亮点度：20%

## 追问问题模板

### 针对"具体性"不足

```
能具体说说吗？比如：
- 你负责哪个产品/模块？
- 使用什么技术栈？
- 团队有多少人？
```

### 针对"量化程度"不足

```
能用数字说明吗？比如：
- 用户量/数据量是多少？
- 提升了多少百分比？
- 节省了多少时间/成本？
```

### 针对"完整性"不足

```
还有几个信息想了解：
- 这个项目的背景是什么？
- 你的具体角色是什么？
- 最终取得了什么成果？
```

### 针对"亮点度"不足

```
有没有特别值得说的：
- 解决过什么难题？
- 有没有创新方案？
- 获得过什么认可或奖励？
```

## 特殊情况处理

### 用户表示"不记得了"

**策略**：
1. 不要追问
2. 提供替代性问题
3. 允许跳过

**示例**：
```
用户：具体数字不记得了
助手：没关系！可以提供一个大概的范围吗？比如是万级、十万级还是百万级？
```

### 用户表示"不方便说"

**策略**：
1. 理解并接受
2. 换一个角度提问
3. 允许跳过

**示例**：
```
用户：薪资数据不方便透露
助手：理解！那可以说说其他方面的成果吗？比如技术改进、效率提升等
```

### 用户回答过于冗长

**策略**：
1. 总结关键信息
2. 确认理解是否正确
3. 引导聚焦核心

**示例**：
```
用户：（说了500字的工作描述）
助手：让我确认一下核心信息：你负责XX系统，带领X人团队，主要成果是XX，对吗？
```

## 质量评估检查清单

在评估每个回答时，检查以下问题：

- [ ] 是否有具体的产品/项目名称？
- [ ] 是否有具体的技术栈？
- [ ] 是否有明确的角色定位？
- [ ] 是否有量化数据（数字/百分比）？
- [ ] 是否有时间范围？
- [ ] 是否有成果描述？
- [ ] 是否有独特亮点？
- [ ] 是否回答了问题的所有要点？

**评分计算**：
- 每个 [ ] 选中的加1分
- 最高8分，根据回答质量微调到10分制
